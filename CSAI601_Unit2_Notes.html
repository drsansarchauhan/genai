8.1 What Is Prompt Engineering?

A prompt is any text you send to a language model as input. Prompt engineering is the systematic practice of designing, iterating, and optimising those inputs to reliably produce the desired outputs — without changing the model's weights.

Core InsightAn LLM is a frozen function — you cannot change its weights at inference time. The only lever you have is the input. Prompt engineering is the art of pulling that lever skillfully.

The Prompt-as-Program Metaphor

Prompt ComponentProgramming AnalogyExample
InstructionFunction signature / docstring"Summarise the following article in 3 bullet points."
ContextGlobal variables / imports"You are an expert in medical research. Audience: general public."
Input dataFunction arguments"Article: {article_text}"
Output formatReturn type / schema"Respond in JSON with keys: summary, key_points, sentiment."

8.2 Why Prompt Engineering Matters

The same model (Claude Sonnet, GPT-4, etc.) can produce wildly different results depending on how the prompt is written. Studies show that prompt variations can cause accuracy swings of 30–50 percentage points on benchmark tasks — larger than the gap between model generations.

Poor Prompt

  • Vague — 'Tell me about transformers'
  • No audience or depth context
  • No output format
  • Result: generic Wikipedia answer
  • Inconsistent across runs

Engineered Prompt

  • Explain transformer self-attention
  • For a 3rd-year CS student
  • One analogy + one equation
  • Under 200 words
  • Result: precise, consistent, actionable

8.4 Anatomy of an Effective Prompt

Instruction

What task should Claude perform? Be explicit: classify, summarise, translate, extract. Never vague.

Context

Background info: who you are, the domain, constraints, relevant history, audience.

Input Data

The text, code, or data to operate on. Delimit with XML tags: <document>...</document>

Output Format

Exact schema, length limit, tone, language. Show a template if needed.

Anti-Patterns to Avoid
  • Vague task: 'Help me with this' → no outcome defined
  • Missing format: 'Give me a list' → count and structure unspecified
  • No audience: Claude defaults to generic register
  • Overloaded task: asking for 5 outputs in one prompt
  • Context dump: pasting 10,000 tokens without prioritising key facts

Zero-Shot & Few-Shot Prompting

Learn when to rely on pre-training knowledge alone vs. when to steer with demonstrations. Master the full spectrum from zero-shot to many-shot, and apply bootstrap techniques when data is scarce.
⏰ 1 Hour📚 CO2🎯 Core Technique

🎯 Learning Objectives

  • Explain the difference between zero-shot, one-shot, few-shot, and many-shot prompting
  • Write zero-shot prompts that leave no room for misinterpretation
  • Design high-quality few-shot demonstrations: coverage, realism, consistency
  • Apply many-shot prompting using Claude's 200K context window

9.1 The Prompting Spectrum

Before models see a single example from you, they already carry billions of training patterns inside their weights. Zero-shot prompting exploits that prior knowledge directly; few-shot prompting steers it with handpicked demonstrations embedded in the prompt itself.

Zero-Shot

No examples. Pure instruction. Works when the task is unambiguous and well-covered by training data.

One-Shot

A single demonstration to anchor format and tone. Best when you have exactly one good example.

Few-Shot (2–8)

Multiple input-output pairs. Model generalises the pattern to novel inputs. Most common in production.

Many-Shot (20+)

Long-context window packed with examples. Approaches fine-tuning quality for narrow tasks.

9.2 Zero-Shot Prompting

Zero-shot works because LLMs generalise from pre-training. The craft is in framing the task completely and unambiguously in a single instruction block.

Five Elements of a Strong Zero-Shot Prompt

ElementWeak VersionStrong Version
Task nameTell me about thisSummarise this review in exactly 2 sentences
Output formatGive keywordsList 5 keywords, comma-separated, all lowercase
AudienceExplain neural netsExplain neural nets to a high-school student with no maths background
ConstraintsWrite an emailWrite a 3-sentence follow-up email; professional tone; no jargon
RoleCheck my codeAct as a senior Python security engineer. Find vulnerabilities.
When Zero-Shot Fails
  • Task demands a precise proprietary schema (custom JSON, specific table layout)
  • Domain is highly specialised and under-represented in training data
  • Desired style is idiosyncratic (brand voice, specific author cadence)
  • Multi-step reasoning with exact intermediate steps is required

In these situations, provide examples (few-shot) or explicit reasoning steps (chain-of-thought).

9.3 Few-Shot Prompting

Few-shot prompting embeds worked examples as Human/Assistant turns before the actual question. Claude learns the format, tone, and reasoning style from the demonstrations and applies that pattern to the new input.

Example: Support Ticket Classification

System: Classify support tickets into: Billing | Technical | Shipping | Other.

H: "My invoice shows double charge."    A: Billing
H: "App crashes on iOS 17."             A: Technical
H: "Package not delivered in 10 days."  A: Shipping

H: "I want to cancel my subscription."
A:

Claude outputs: Billing — correctly generalising from 3 demonstrations.

Rules for Designing Good Demonstrations

RuleWhy It Matters
Cover the output spaceShow at least one example per class/label — Claude learns all valid outputs
Use realistic inputsToy examples generalise poorly to real data distributions
Keep format perfectly consistentAny deviation in example format leaks into Claude's output
Order by difficultyEasy cases first, edge cases last (closest to the real query)
Label accuratelyOne wrong label hurts more than having no examples at all
Match the domainExamples from a different domain confuse rather than help

9.4 Format Priming

Use demonstrations purely to lock in output structure, not to teach the task semantics. This is useful when Claude already understands the task but needs to match a specific format.

Review: "Good laptop but fan is loud."
Output:
  summary: decent performance
  pros: [portability, battery]
  cons: [fan noise]
  score: 6/10

Review: "Best phone I have owned. Camera is stunning."
Output:
  summary: excellent all-rounder
  pros: [camera, display, build]
  cons: []
  score: 9/10

Review: {{USER_REVIEW}}
Output:

The demos teach format here, not task semantics — Claude already knows sentiment analysis.

9.5 Many-Shot & Bootstrap

Claude supports context windows up to 200K tokens, enabling 50–500 demonstrations in a single prompt. Anthropic research shows many-shot can match or exceed fine-tuning on classification and extraction tasks — without any gradient updates.

Bootstrap Technique — When You Have No DataStep 1: Ask Claude to generate 5–10 realistic examples of input + ideal output.
Step 2: Review and correct the generated examples.
Step 3: Use the curated examples as your few-shot demonstrations.
This works surprisingly well and lets you iterate on demonstration quality before committing to a labelled dataset.

9.6 Choosing: Zero-Shot vs Few-Shot

FactorLean Zero-ShotLean Few-Shot
Task familiarityCommon, well-defined taskNovel or specialised task
Output formatFree-form prose is fineStrict schema required
Token budgetTight (cost sensitive)Budget allows demonstration overhead
Data availabilityNo labelled examplesHave 2–20 quality examples
Iteration phaseRapid prototypingProduction system tuning
Practical WorkflowStart zero-shot. If quality is insufficient after 3 iterations, add 2–3 examples (few-shot). If still insufficient, move to chain-of-thought (L10) or fine-tuning.

Chain-of-Thought & Reasoning Prompts

⏰ 1 Hour📚 CO2🎯 Advanced Technique

🎯 Learning Objectives

  • Use the Extended Thinking API for Claude's hardest reasoning tasks

10.1 Why Reasoning Prompts Matter

Large language models are next-token predictors. When asked a multi-step question directly, they produce an answer token by token without any explicit reasoning buffer — leading to brittle results on problems that require arithmetic, logical deduction, or sequential planning.

Key Finding (Wei et al., 2022)Adding 'Let's think step by step.' to a prompt improved GPT-3 accuracy on grade-school math from 17.7% to 78.7%. The same principle applies to Claude — and the improvement is largest on the hardest problems.

10.2 Chain-of-Thought Variants

Zero-Shot CoT

Append 'Think step by step' — no examples needed. Activates latent reasoning capability in any large LLM.

Few-Shot CoT

Provide demonstrations that include full reasoning chains, not just final answers. Model mimics reasoning style.

Self-Consistency

Sample N reasoning paths, take majority vote on the final answer. Reduces variance by 5–15%.

Extended Thinking

Claude 3.7+ allocates scratchpad tokens before answering. Best for hardest math, code, planning.

10.3 Zero-Shot CoT in Practice

Effective Trigger Phrases

PhraseBest Use
Let's think step by step.General reasoning, maths, logic puzzles — most widely tested
Walk me through your reasoning.Analytical tasks, diagnoses, recommendations
First, identify all relevant facts. Then solve.Structured problem-solving with known unknowns
Before answering, list all assumptions.Ambiguous problems with hidden constraints
Show your work.Calculations, proofs, derivations

Before vs After CoT

WITHOUT CoT:
Q: Train A leaves Delhi at 9am at 80 km/h. Train B leaves Mumbai at 10am
   at 100 km/h toward Delhi. Distance 1400 km. When do they meet?
A: 2:30pm  [WRONG]

WITH 'Let's think step by step.':
A: 1. In the first hour, Train A covers 80 km. Gap at 10am = 1320 km.
   2. Both moving toward each other: closing speed = 80+100 = 180 km/h
   3. Time to close 1320 km: 1320/180 = 7.33 h = 7 hr 20 min
   4. 10am + 7h 20min = 5:20pm
   Answer: 5:20pm  [CORRECT]

The reasoning chain also makes errors visible and correctable — you can see exactly where logic failed.

10.4 Few-Shot CoT

Provide demonstrations where the reasoning chain is shown explicitly. Claude will mimic your reasoning style and depth — so match chain complexity to your actual task.

Q: Roger has 5 balls. He buys 2 cans of 3 balls each. How many balls?
A: Roger starts with 5. He buys 2x3=6 more. Total: 5+6=11. Answer: 11.

Q: A juggler has 16 balls. Half are golf balls. Half of golf balls are blue.
   How many blue golf balls?
A: Total: 16. Golf balls: 16/2=8. Blue golf balls: 8/2=4. Answer: 4.

Q: [YOUR QUESTION]
A:

Design tip: make reasoning chains match the complexity of your target tasks. Simple 2-step demos don't transfer to 6-step problems.

10.5 Self-Consistency Decoding

Instead of taking a single reasoning path, generate multiple chains (temperature > 0) and take a majority vote on the final answer. Typically improves accuracy by 5–15% on arithmetic and logical reasoning benchmarks.

import anthropic, re
from collections import Counter

client = anthropic.Anthropic()
question = 'What is 17% of 340? Let\'s think step by step.'

answers = []
for _ in range(5):  # sample 5 reasoning paths
    r = client.messages.create(
        model='claude-opus-4-5',
        max_tokens=300,
        messages=[{'role':'user','content': question}]
    )
    nums = re.findall(r'\d+\.?\d*', r.content[0].text)
    if nums: answers.append(nums[-1])

final = Counter(answers).most_common(1)[0][0]
print('Majority answer:', final)  # 57.8

Sampling 5 paths and voting reduces the chance of a confident-but-wrong single answer.

10.6 Extended Thinking API

Claude 3.7 Sonnet exposes a thinking block — allocated compute tokens used for internal scratchpad reasoning before the final response is generated. The scratchpad is visible in the response for debugging.

response = client.messages.create(
    model='claude-sonnet-4-5',
    max_tokens=16000,
    thinking={
        'type': 'enabled',
        'budget_tokens': 10000   # scratchpad allocation
    },
    messages=[{'role':'user','content':'Solve: ...'}]
)

for block in response.content:
    if block.type == 'thinking':
        print('Scratchpad:', block.thinking)
    elif block.type == 'text':
        print('Final answer:', block.text)

Cost: ~3x token overhead. Use for complex math, competitive coding, long-horizon planning, scientific reasoning.

11.1 Why Role Assignment Works

Assigning a role to Claude does more than change its tone — it activates a cluster of knowledge, terminology, values, and communication patterns associated with that role in Claude's training. A senior cardiologist, a startup founder, and a high-school teacher will explain 'cardiac risk' in structurally different ways, and so will Claude when primed with each role.

Professional Roles

Doctor, lawyer, engineer, teacher — activates domain vocabulary and epistemic standards.

Creative Roles

Novelist, screenwriter, copywriter — unlocks narrative voice and creative latitude.

Functional Roles

Critic, devil's advocate, fact-checker — shapes the evaluative lens Claude applies.

Hybrid Roles

Stack roles for precision: 'UX designer with a behavioural psychology background.'

11.2 Building a Full Persona

A persona is richer than a role. It combines identity, expertise, personality, communication style, and constraints into a coherent character Claude maintains across a conversation.

You are Priya, a senior data scientist at a B2B SaaS company.

Identity:
- 8 years experience in ML, specialising in churn prediction
- Direct communication style — no filler phrases
- Uses concrete numbers over vague adjectives

Expertise:
- Python, SQL, scikit-learn, XGBoost, Spark
- Familiar with Salesforce, Mixpanel, Amplitude

Communication rules:
- Always ask for business context before going technical
- When uncertain, say so explicitly rather than guessing
- Offer 2-3 options with trade-offs, never a single answer

Do NOT:
- Use marketing language ('game-changing', 'synergy')
- Recommend paid tools when free alternatives exist

Persona = Identity + Expertise + Communication Rules + Constraints. Specify all four layers.

11.3 Five Tone Dimensions

Layered Tone Specification

Write customer support responses with:
- Tone: warm, empathetic, professional
- Formality: semi-formal (no slang; contractions allowed)
- Length: 3-5 sentences maximum

Structure (always follow):
  Sentence 1: acknowledge the specific issue
  Sentences 2-3: provide the solution steps
  Final sentence: warm closing with next step

Forbidden phrases:
- 'I understand your frustration' (overused)
- 'No problem!' (too casual)
- 'Going forward' (corporate jargon)

Forbidden phrase lists are often more robust than positive tone descriptions.

12.1 Why Structured Output Matters in Production

In production applications, Claude's output is rarely read by a human directly — it feeds downstream systems: databases, APIs, UI renderers, workflow automations. If the output format is inconsistent, the whole pipeline breaks regardless of content quality.

The Golden RuleAny system that parses Claude's output programmatically must specify the output format explicitly in the prompt. Never rely on implicit formatting — what works 95% of the time fails 5% and breaks your application.

12.2 JSON Output — Three Techniques

TechniqueHowReliability
Explicit schema instructionPaste the JSON schema in the prompt; say 'Return ONLY valid JSON'Good — 90–95%
Seed the assistant turnStart the assistant turn with '{' to force JSON openingBetter — 95–98%
System prompt modeSystem: 'You output only valid JSON. No extra text.'Best — 97–99%
Validation + retry (recommended)Catch JSONDecodeError; feed error back; retry up to 3 timesNear 100%

JSON Output with Schema

System: You output only valid JSON. No extra text, no code fences.

Schema:
{
  "name": string,
  "date": "YYYY-MM-DD",
  "amount": number,
  "currency": "USD"|"EUR"|"INR"
}

Text: "Received payment of Rs.5,000 from Amit on 12 March 2025."

Expected output:
{"name":"Amit","date":"2025-03-12","amount":5000,"currency":"INR"}

Always validate immediately with json.loads() — never assume success.

12.3 Pydantic Validation + Retry Pattern

from pydantic import BaseModel, ValidationError
import json, anthropic

class Ticket(BaseModel):
    category: str
    priority: int  # 1-5
    summary: str
    tags: list[str]

client = anthropic.Anthropic()

def extract_ticket(text: str, retries: int = 3) -> Ticket:
    for attempt in range(retries):
        r = client.messages.create(
            model='claude-opus-4-5', max_tokens=300,
            system='Output only valid JSON matching the Ticket schema.',
            messages=[{'role':'user','content': text}]
        )
        try:
            return Ticket(**json.loads(r.content[0].text))
        except (json.JSONDecodeError, ValidationError) as e:
            if attempt == retries - 1: raise
            text = f'Error: {e}. Fix and retry.\n{text}'

Feeding the exact error message back to Claude lets it self-correct on retry. Reaches near 100% compliance.

12.4 XML and Custom Formats

XML Output (for nested/hierarchical data)

Return in this exact XML structure:
<analysis>
  <sentiment>positive|negative|neutral</sentiment>
  <confidence>0.0 to 1.0</confidence>
  <reasons>
    <reason>TEXT</reason>
  </reasons>

XML is ideal for nested structures. Claude has a natural affinity for XML from its training.

Return each finding on a new line in this format:
SEVERITY|COMPONENT|DESCRIPTION

Example:
HIGH|AuthService|JWT tokens are not expiring
MEDIUM|Database|Connection pool not releasing

Pipe-delimited formats are easy to parse line-by-line and work well for ETL pipelines.

13.1 Understanding the Context Window

The context window is the total number of tokens Claude can see at once — including the system prompt, all conversation history, and the current message. Claude Sonnet and Opus 4 support 200K tokens, roughly equivalent to a 150,000-word novel or 600 pages of dense text.

ModelContext WindowApprox. Pages
Claude Haiku 4.5200K tokens~600 pages
Claude Sonnet 4200K tokens~600 pages
Claude Opus 4200K tokens~600 pages
The 'Lost in the Middle' ProblemResearch shows LLMs recall information at the beginning and end of the context most reliably. Content in the middle of a very long context is retrieved less accurately. Design implication: place critical instructions in the system prompt AND just before the question.

13.2 Context Management Strategies

Sliding Window

Keep only the N most recent turns. Oldest turns are dropped. Simple but loses early context.

Summarisation

Periodically compress history into a summary, replace raw turns. Preserves key facts cheaply.

RAG

Store turns in a vector DB. Retrieve the most relevant turns for each new message.

Hierarchical Memory

Working memory (recent) + episodic (summaries) + semantic (facts) managed separately.

13.3 Conversation Summarisation

import anthropic
client = anthropic.Anthropic()
MAX_TURNS = 10

def summarise_history(history: list) -> str:
    r = client.messages.create(
        model='claude-haiku-4-5-20251001',
        max_tokens=500,
        messages=[*history,
            {'role':'user',
             'content':'Summarise: decisions made, facts established, open questions.'}]
    )
    return r.content[0].text

def chat(history: list, user_msg: str) -> tuple:
    if len(history) > MAX_TURNS * 2:
        summary = summarise_history(history)
        history = [{'role':'assistant','content': f'[Summary]\n{summary}'}]
    history.append({'role':'user','content': user_msg})
    r = client.messages.create(model='claude-opus-4-5', max_tokens=1000, messages=history)
    reply = r.content[0].text
    history.append({'role':'assistant','content': reply})
    return reply, history

Cost tip: use cheap Haiku for summarisation; use Opus/Sonnet for the final responses.

Map-Reduce Summarisation

def map_reduce_summarise(text: str, chunk_size: int = 50000) -> str:
    words = text.split()
    chunks = [' '.join(words[i:i+chunk_size]) for i in range(0, len(words), chunk_size)]

    # Map: summarise each chunk with Haiku
    summaries = []
    for chunk in chunks:
        r = client.messages.create(
            model='claude-haiku-4-5-20251001', max_tokens=500,
            messages=[{'role':'user','content':f'Summarise:\n{chunk}'}]
        )
        summaries.append(r.content[0].text)

    # Reduce: combine summaries with Opus
    r = client.messages.create(
        model='claude-opus-4-5', max_tokens=1000,
        messages=[{'role':'user','content': 'Combine these summaries:\n' + '\n\n'.join(summaries)}]
    )
    return r.content[0].text

Use Haiku for the map phase (cheap/fast); Opus or Sonnet for the reduce phase (quality).

14.1 Why Systematic Evaluation Matters

A prompt that works on 3 test cases may fail on the 4th. A prompt improved by intuition may actually degrade on edge cases you did not test. Systematic prompt evaluation — measuring quality across a diverse test set — is what separates ad-hoc prompting from reliable prompt engineering.

Goodhart's Law for Prompts'When a measure becomes a target, it ceases to be a good measure.' Optimising a prompt on a small test set leads to overfitting. Always hold out a validation set that you never use during iteration.

14.3 LLM-as-Judge

Using Claude to evaluate Claude's outputs is surprisingly effective and scales to tasks where labelled ground truth is unavailable. The key is a well-designed evaluation rubric prompt.

EVAL_PROMPT = (
    'You are an expert evaluator.\n'
    'Task: {task}\nResponse: {response}\n\n'
    'Score each dimension 1 (poor) to 5 (excellent):\n'
    'accuracy, completeness, clarity, format.\n'
    'Return ONLY JSON: '
    '{"accuracy":N,"completeness":N,"clarity":N,"format":N,'
    '"overall":N,"critique":"one sentence"}'
)

def evaluate(task: str, response: str) -> dict:
    r = client.messages.create(
        model='claude-haiku-4-5-20251001',
        max_tokens=200,
        messages=[{'role':'user','content': EVAL_PROMPT.format(task=task,response=response)}]
    )
    return json.loads(r.content[0].text)

Use Haiku as judge — cheap, fast, and surprisingly accurate on well-defined rubrics.

14.4 A/B Prompt Testing

import statistics

def compare_prompts(prompt_a, prompt_b, test_cases, n_trials=3):
    results = {'A': [], 'B': []}
    for case in test_cases:
        for prompt, label in [(prompt_a,'A'),(prompt_b,'B')]:
            scores = []
            for _ in range(n_trials):
                r = client.messages.create(
                    model='claude-opus-4-5', max_tokens=500,
                    messages=[{'role':'user','content':prompt.format(**case)}]
                )
                s = evaluate(case['task'], r.content[0].text)
                scores.append(s['overall'])
            results[label].append(statistics.mean(scores))

    a = statistics.mean(results['A']); b = statistics.mean(results['B'])
    return {'A_mean':a, 'B_mean':b, 'winner':'A' if a > b else 'B'}

Always compare on the SAME test set. Multiple trials reduce variance from model non-determinism.
